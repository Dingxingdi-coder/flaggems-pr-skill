# Operator Quality Gate

本文件描述 Coding Agent 在提交前必须做的人工审查和可修复问题。`check_operator.py` 只能覆盖一部分静态规则；弱模型生成的算子必须经过本 gate 后再提交。

## 1. 审查输入

对每个 `<op>` 至少阅读这些文件：

- `.worktrees/gen-<op>/src/flag_gems/ops/<module>.py`
- `.worktrees/gen-<op>/tests/<test_file>.py`
- `.worktrees/gen-<op>/benchmark/<bench_file>.py`
- `.worktrees/gen-<op>/src/flag_gems/ops/__init__.py`
- `.worktrees/gen-<op>/src/flag_gems/__init__.py`
- `.worktrees/gen-<op>/conf/operators.yaml`
- 主仓库中同类型、已合入的 FlagGems 算子作为风格参考

生成代码默认不可信。若发现问题，先修改 worktree；提取后的 PR 分支只保留从修复后 worktree 复制出的实现和脚本产生的机械调整。

## 2. 必须修复或放弃的红线

出现以下任一情况，不得提交 PR：

- kernel 核心路径调用 `torch.<op>`、`torch.nn.functional.*`、`torch.matmul/mm/softmax/add` 等 torch fallback 来完成目标计算。
- wrapper 只是调用 PyTorch 或递归调用目标算子，没有真正使用 Triton/FlagGems 低层实现。
- `@triton.jit` kernel 存在但没有被 wrapper 调用，或只是装饰性死代码。
- test 没有验证目标 dispatch 路径，或者 golden reference 未使用 `utils.to_reference()`。
- benchmark 中 torch wrapper 与 gems wrapper 计算量不等价，或为了跑通 benchmark 减少了 gems 侧工作量。
- benchmark `--level core` 为 0 cases。
- dtype 覆盖与 PyTorch 支持情况不一致；不支持的 dtype 没有在 wrapper 中明确拒绝或在测试中说明。
- yaml `id`、pytest mark、benchmark `op_name`、文件名、`_FULL_CONFIG` 不一致。
- 导出了 `_out`、inplace 或 overload 变体，但没有对应 yaml/test/benchmark 覆盖。
- 会修改上游已有测试函数或提交不属于当前算子的文件。
- 平均 speedup < 0.8 且无法通过合理优化改善。

## 3. Kernel 规则

Kernel 文件首行必须是 `# Generated by KernelGen: https://github.com/flagos-ai/KernelGen`。使用 `logging`，禁止 `print()`。函数名用 snake_case，类名 PascalCase，常量 UPPER_CASE。删除重复函数、死代码、无意义 alias wrapper、未被调用的 `@use_tl_extra` 桩函数。

核心计算必须由 Triton kernel、`flag_gems.utils.pointwise_dynamic`、FlagGems 已实现算子，或其他明确的 FlagGems 低层工具完成。允许的 torch 用途仅限辅助：分配输出 tensor、读取 shape/stride/dtype/device、reshape/view/permute/contiguous、类型推断、复数 view 等。不得用 torch 做核心数学计算。

如果限制 dtype，例如只支持 fp32，wrapper 中要有 assert/raise 明确拒绝不支持 dtype；测试和 benchmark 中硬编码 `[torch.float32]` 时必须有注释说明 PyTorch 或 kernel 的限制。不要用 `is_cuda` 等平台 hardcode，只检查 tensor device 一致性。

硬编码 BLOCK、num_warps、shape 上限等超参必须有注释说明。可 autotune 的配置应放入 `tune_configs.yaml` 或使用项目已有配置方式。使用 libdevice 时优先检查上游跨后端规范；如 reviewer 关注兼容性，改用 `tl_extra_shim`。

## 4. Accuracy test 规则

测试文件使用：

```python
import pytest
import torch
import flag_gems
from . import accuracy_utils as utils
```

pytest mark 使用 op id：前导 `_` 去掉，尾部 `_` 保留，多重载后缀与 yaml `id` 完全一致。测试函数名使用 `test_<op_id>` 或每个 overload 一个 `test_<id>`，不要保留 `test_accuracy_` 前缀。

Golden reference 使用 `utils.to_reference()`。浮点结果用 `utils.gems_assert_close(res, ref, dtype, ...)`，禁止传 `rtol`；精确/布尔/索引用 `utils.gems_assert_equal`。NaN 比较用 `equal_nan=True`，不要手写 NaN 过滤。不要显式 `.cpu()` 或 `.to(torch.float64)` 构造 reference。

dtype 要先验证 PyTorch reference 是否支持。默认使用 `utils.FLOAT_DTYPES` 等公共常量；PyTorch 不支持 fp16/bf16 的 linalg、special、igamma、cdist、仿射变换等算子，可以缩小 dtype，但要写注释并让 wrapper 拒绝不支持 dtype。

概率算子（dropout、bernoulli、rand 等）用统计方法验证 mean/variance 或分布特征，不做逐元素精确比较。不要保留 `QUICK_MODE`、vendor-specific 分支或调试 `print()`。

## 5. Benchmark 规则

benchmark 文件使用 `pytest` 和 `benchmark/base.py` 的封装，不允许自定义框架。mark 与 yaml `id` 一致，`op_name` 与 mark 一致。默认 dtype 用 `consts.FLOAT_DTYPES`；若只支持有限 dtype，硬编码必须有注释且不要导入未使用的 `consts`。

一元/二元 pointwise 优先用 `base.UnaryPointwiseBenchmark` / `base.BinaryPointwiseBenchmark`。Reduction 可用 `base.UnaryReductionBenchmark`。非 pointwise、自定义输入、backward、out/inplace 或多输入结构复杂的算子，优先保留并修复 worktree 中的 Benchmark 子类；必须保证 `--level core` 加载 shape 后仍能生成有效输入。

`base.GenericBenchmark(input_fn=...)` 只适用于 shape tuple 能直接驱动输入生成的简单场景，并且必须通过 `pytest benchmark/test_<op>.py -s --level core` 证明非 0 cases。不要用 GenericBenchmark 替代复杂 Benchmark 来掩盖 `set_shapes` 或输入结构问题。

非 pointwise Benchmark 子类通常需要覆盖 `set_shapes` 或正确使用 `core_shapes.yaml`，使 CI 的 `--level core` 不会把自定义输入结构覆盖坏。`get_input_iter` 中的 hardcoded shapes 需要注释；更好的做法是放进 `core_shapes.yaml`。

Benchmark 必须公平：torch op 和 gems op 参数一致、计算量一致、同步方式一致。backward benchmark 只 benchmark backward，不把 forward 额外计入其中，除非 torch 和 gems 两侧完全对等。

## 6. YAML 与注册规则

`conf/operators.yaml` 按 `id` 字母序插入。新 KernelGen 算子 labels 必须包含 `aten` 和 `KernelGen`，按类型补充 `pointwise`、`reduction`、`nn.functional` 等；`kind` 用项目已有类别；stage 按当前任务约定使用 `alpha: '5.1'`。`description` 应来自 PyTorch 文档或准确的一句话说明。

`src/flag_gems/ops/__init__.py` 添加 import 和 `__all__`，按字母序，前导下划线模块排在普通模块前。`src/flag_gems/__init__.py` 的 `_FULL_CONFIG` 使用 aten 名；overload 保留点号，如 `("norm.Scalar", norm_scalar)`；前导下划线的 aten name 保留下划线。

多变体一致性：yaml 条目数、pytest mark 数、benchmark 函数数、`__all__` 导出数、`_FULL_CONFIG` 条目数必须能解释为同一组 dispatch 路径。未注册、未测试的导出函数必须删除。

## 7. 提交前自问

- 这个算子离开 PyTorch fallback 还能计算正确吗？
- 测试覆盖的是 reviewer 会关心的真实 dispatch、dtype、shape、edge case 吗？
- benchmark 的每一行数据都来自 `--level core`，且 torch/gems 计算量一致吗？
- reviewer 常见问题中关于命名、mark、dtype、benchmark、libdevice、append mode 的条目都排查了吗？
- PR 分支的 diff 是否只有目标算子的六类文件？
