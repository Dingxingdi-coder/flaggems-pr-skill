# Immutable FlagGems PR Spec

This file is the only long-lived reference every subagent must obey. Agent task files may be short, but they cannot override this spec.

## PR scope

- One PR contains exactly one aten operator family requested by the user. Unrelated operators, opportunistic cleanups, and broad refactors are forbidden.
- Branch name is `pr/<op>`, where `<op>` is the normalized operator name returned by `operator_registry.py lookup`.
- Base branch is the user-provided upstream remote/base branch. Do not rebase and do not cherry-pick generated commits.
- Push remote, fork GitHub repo, upstream GitHub repo, tested-on text, repo paths, worktree root, norm xlsx path, GPU IDs, and domestic GPU summary root are all user/context inputs, never hardcoded.

## Naming

- First map the raw operator name through `operator_registry.py lookup`.
- Use `scripts/op_naming.py` to derive module name and public `op_id`.
- Leading underscore is removed only from pytest mark, yaml `id`, test filename, benchmark filename, test function, and benchmark `op_name`.
- Leading underscore is preserved in kernel filename, function name, import, `__all__`, `_FULL_CONFIG` aten name, yaml `for`, and actual torch call name.
- Trailing underscore for inplace is preserved everywhere.
- For overloads, yaml `id` = pytest mark = benchmark `op_name`; yaml `for` keeps aten dotted names such as `norm.Scalar`.

## Worktree-first implementation

- Generated code is untrusted. Review and repair `<worktree_root>/gen-<op>` first.
- PR branch code must be extracted from the repaired worktree. Do not bypass extraction by rewriting implementation/test/benchmark directly in the PR branch.
- Pure formatting, unused import removal, yaml description wording, and PR body edits may happen on the PR branch.
- Do not delete valuable generated comments unless they are wrong or misleading.

## Kernel requirements

- Kernel file starts with `# Generated by KernelGen: https://github.com/flagos-ai/KernelGen`.
- Core computation must use Triton, `pointwise_dynamic`, FlagGems low-level utilities, or existing FlagGems ops. PyTorch fallback for target computation is forbidden.
- Forbidden in core paths: `torch.<target>`, `torch.nn.functional.*`, `torch.matmul`, `torch.mm`, `torch.add`, `torch.softmax`, or equivalent torch implementation.
- Allowed torch uses are auxiliary: output allocation, shape/stride/dtype/device inspection, view/reshape/permute/contiguous, type inference, and complex view helpers.
- No `print()`. Use logging.
- Remove duplicate functions, dead exports, meaningless alias wrappers, and unused `@use_tl_extra` stubs.
- Hardcoded block sizes, shapes, dtype limitations, and platform restrictions need comments. Unsupported dtypes should be rejected by wrapper checks.

## Accuracy tests

- Use relative import: `from . import accuracy_utils as utils`.
- Use `utils.to_reference()` for PyTorch reference inputs/outputs.
- Use `utils.gems_assert_close` for floating results and `utils.gems_assert_equal` for exact results. Never pass `rtol` to `gems_assert_close`.
- NaN comparisons use `equal_nan=True`.
- Prefer shared dtype/shape constants. If dtype is restricted, first verify PyTorch reference support and add comments.
- Remove `QUICK_MODE`, vendor-specific branches, debug output, and unnecessary skips.
- Probability operators use statistical checks rather than elementwise equality.
- Do not modify upstream existing tests except append-mode additions for the target op.

## Benchmark

- Use pytest and `benchmark/base.py` wrappers. Standalone/custom benchmark frameworks are forbidden.
- Benchmark mark and `op_name` equal yaml `id`.
- `--level core -s` is mandatory for PR data.
- Benchmark must produce nonzero parsed cases.
- Torch and Gems sides must have equivalent arguments, computation amount, forward/backward boundary, and dtype coverage.
- Pointwise may use standard unary/binary wrappers. Non-pointwise or complex input ops should keep a Benchmark subclass with `set_shapes`/`get_input_iter` that survives `--level core`.
- Mean speedup below `0.8x` blocks submission unless the user explicitly decides to abandon the operator.

## YAML and registration

- Modified files should normally be only: kernel, `src/flag_gems/ops/__init__.py`, `src/flag_gems/__init__.py`, test, benchmark, `conf/operators.yaml`.
- `operators.yaml` entry must include accurate `description`, correct `for`, labels including `aten` and `KernelGen`, valid `kind`, and expected stage from context.
- Import lists and `__all__` remain sorted according to project style.
- `_FULL_CONFIG` uses actual aten dispatch names; overloads keep dotted names.
- Exported functions must be registered and covered by tests/benchmarks, or removed.

## Validation and PR body

- Required gates: preflight, worktree quality gate, worktree tests/benchmark, extraction, `check_operator.py --strict`, overload consistency, pre-commit, PR-branch accuracy test, PR-branch benchmark.
- PR body is English and includes Summary/Description, Testing/Correctness, Performance, Tested-on, Multi-backend Testing, Files Changed.
- Performance data must come from actual `--level core` benchmark output. Multi-backend data must come from provided logs/summaries or be marked `N/A`; never fabricate.
- Do not create PRs with missing speedup, zero cases, failed tests, or incomplete required PR body sections.

## Git prohibitions

- Never use `git add -A` or `git add .`.
- Never use `git cherry-pick` or `git rebase`.
- Never include `Co-Authored-By` in commit messages.
- Never bypass failing scripts or skip tests/benchmark for a real PR.
